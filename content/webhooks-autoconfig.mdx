# Webhooks AutoConfig

Webhooks AutoConfig is a declarative, code-first way to configure webhook endpoints, using the Svix SDK.

Key benefits for your customers:
- Define the webhook configuration in code (rather than both code and the Application Portal UI).
- Automatically update URL, event types and other endpoint settings.
- Signature verification is supported out of the box, no additional secrets to wire up.

## How it works

When AutoConfig is enabled in your environment, your users will be able to choose it as an option when creating an endpoint in the Application Portal.
The portal will issue an *AutoConfig token*, which can be used to configure the endpoint in code.

![AutoConfig option in the Application Portal](/img/autoconfig/autoconfig-create.png)

Using the Svix SDK `AutoConfig` client, the consumer specifies the endpoint configuration, passes the token and calls `subscribe()`.
`subscribe()` will update the endpoint to match the specified configuration.

Calling `subscribe()` can be done as part of the application startup process, or in CI/CD as part of the deployment process, so the endpoint configuration is always in sync with the latest version of the application code.

For more information on how to receive webhooks using AutoConfig, see [Receiving with Webhooks AutoConfig](/receiving/webhooks-autoconfig).

### AutoConfig Example

<CodeTabs items={["JavaScript","Python","Rust","Go","Java","Kotlin","Ruby","C#","PHP"]}>

<TabItem value="JavaScript">
```js
import { AutoConfig } from "svix";

const webhook = new AutoConfig({
  token: AUTO_CONFIG_TOKEN,
  eventTypes: ["invoice.paid", "user.created"],
  url: "https://api.us.example.com/webhooks/acme",
});

await webhook.subscribe();
```

### Updating the endpoint configuration

To update the endpoint configuration, your customers simply need to change the AutoConfig options and call `subscribe()` again.
No need for any manual changes in the Application Portal UI.

### Signature verification

The same `AutoConfig` instance is also used to verify signatures for incoming webhooks. No need to configure a separate signature verification secret.
